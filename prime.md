description: Prime a terminal with role-specific context
---

Load focused context for a specific terminal role.

## Usage

```
/prime [role]                    # Prime with role context
/prime [role] --audit <topic>    # Prime + generate audit template
```

## Valid Roles

operator, stw-liaison, cto, architect, coordinator, security, toolsmith, devops, product-owner, planner, wireframe-generator, preview-host, wireframe-qa, validator, inspector, author, test-engineer, developer, auditor, qa-lead, tech-writer, docker-captain

## Instructions

### 1. Read Root Context

Read `CLAUDE.md` (~60 lines).

### 2. Load Role File

| Role | File |
|------|------|
| operator | `.claude/roles/operator.md` |
| stw-liaison | `.claude/roles/stw-liaison.md` |
| cto, architect, security, toolsmith, devops, product-owner | `.claude/roles/council.md` |
| planner, wireframe-generator, preview-host, wireframe-qa, validator, inspector | `.claude/roles/wireframe-pipeline.md` |
| developer, test-engineer, auditor | `.claude/roles/implementation.md` |
| coordinator, author, qa-lead, tech-writer | `.claude/roles/support.md` |
| docker-captain | `.claude/roles/docker-captain.md` |

### 3. Load Role-Specific Inventory

| Role | Inventory File |
|------|----------------|
| security | `.claude/inventories/security-touchpoints.md` |
| toolsmith | `.claude/inventories/skill-index.md` |
| architect | `.claude/inventories/dependency-graph.md` |
| devops | `.claude/inventories/workflow-status.md` |
| product-owner | `.claude/inventories/acceptance-criteria.md` |
| planner, wireframe-generator | `.claude/inventories/screen-inventory.md` |
| docker-captain, devops | `.claude/knowledge/docker/DOCKER_SYSTEM_PROMPT.md` |

Other roles: No additional inventory needed.

### 4. Council Check

For council members only, check for pending RFCs:
```bash
grep -l "Status.*voting" docs/interoffice/rfcs/*.md 2>/dev/null | wc -l
```
If >0, report: "[N] RFC(s) awaiting your vote."

### 5. Audit Template (Optional)

If `--audit <topic>` is provided, generate a standardized audit template:

```bash
# Generate audit template
python3 scripts/audit-template.py <role> <topic>

# Or write directly to file
python3 scripts/audit-template.py <role> <topic> --output docs/interoffice/audits/$(date +%Y-%m-%d)-<role>-<topic>.md
```

**Available templates by role:**

| Role | Topics |
|------|--------|
| toolsmith | skills, scripts, automation |
| architect | deps, patterns |
| security | auth, secrets |
| devops | ci, docker |
| auditor | compliance, code |

To see all templates: `python3 scripts/audit-template.py list`

### 6. Output

> "Read [role] context. Ready."

If inventory loaded, also report:
> "Loaded [inventory-name] inventory."

If audit template generated:
> "Generated [topic] audit template at [path]."

Always include this reminder at the end of output:
> **Context low?** Use `/clear` then `/prime [role]` - NOT `/compact`

## DO NOT

- Summarize the repository
- List what you learned
- Read constitution.md unless specifically needed
- Load full .terminal-status.json (only queue items if needed)

## Audit Format Standard

All audits should follow this structure (generated by `scripts/audit-template.py`):

```markdown
# [Role] Audit: [Topic]

**Date**: YYYY-MM-DD
**Author**: [Role]
**Scope**: [Brief scope description]

---

## Executive Summary
Brief overview of findings.

## [Topic-Specific Section]
Tables, lists, or analysis specific to the audit type.

## Recommendations
1. Numbered recommendations
2. ...

## Action Items
- [ ] Checkbox action items
- [ ] ...

---

## Related Documents
- Links to related files
```

This ensures:
- Consistent formatting across all terminals
- Reduced token overhead (no AI formatting decisions)
- Easy parsing for automation scripts
